<objective>
Fix the CLI-resolution and ergonomics failures that made the Phase 15 dogfood run hard to operate from inside the worktree DevFlow itself created (16f, broadened per RESEARCH to every subcommand), plus the two smaller UX footguns (16g):

- **16f — Shared project-root walk-up resolver.** `project_root(project: PathBuf)` only canonicalizes the given path (default `.`) with zero walk-up, so running any subcommand from inside the phase worktree resolves to the worktree — where no `.devflow/` state lives — making `devflow status` report `idle` and `devflow gate approve 15` fail with "no open gate for phase 15". Every subcommand routes through this one function, so a single walk-up fix (loop `Path::parent()` looking for `.devflow/`, mirroring `git rev-parse --show-toplevel`) fixes all call sites at once. A project with no `.devflow/` anywhere must behave exactly as today.
- **16g — Two UX cleanups.** (a) `devflow gate approve 15 ship` silently binds `"ship"` to the trailing `project` positional (clap trailing-positional ambiguity) and fails with "project path does not exist: ship" — fix the GateCmd schema so a misplaced stage is accepted or hinted. (b) `migrate_legacy_state`'s unparsable-state WARN fires on every `status` with no hint — append a `devflow recover --clean` self-service pointer.

Depends on 16-01 (D-09: 16k's terminal-signal fix is what made status's worktree bug isolable) and 16-03 (shares main.rs — sequenced after it).
Purpose: DevFlow is operable from inside its own worktree, and the two most common CLI footguns give actionable errors.
Output: a shared walk-up `project_root`, a fixed GateCmd schema, and a hinted legacy-state WARN — with tests.
</objective>
