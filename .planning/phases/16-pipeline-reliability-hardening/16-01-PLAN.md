<objective>
Fix the worst signal failure of the Phase 15 dogfood run (16k, sequenced FIRST per D-09): the terminal "Ship complete → approve merge?" advance path never merges the feature branch. `hooks_after_ship()` returns only `[VersionBump, BranchCleanup]` — no merge hook is wired at all — so DevFlow emitted `VersionBump ok=true` / `workflow_finished` while PR #7 stayed open, VersionBump bumped the PRIMARY checkout to a nonsense version (1.2.0→1.2.183) against the wrong tree, and BranchCleanup reported `ok=true` on a no-op.

This plan wires the existing, already-tested `GitFlow::feature_finish` merge primitive into the terminal path via a new `Hook::Merge`, ordered FIRST so VersionBump computes against post-merge `develop`; makes the merge idempotent/PR-safe (a safe no-op when the branch is already merged or absent, per RESEARCH Open Question 1); emits a truthful `merged:bool` signal so hook success is no longer unconditional; and cleans the bogus CHANGELOG entries the pre-fix mechanism already wrote into this repo.

Purpose: every other reliability item in this phase assumes the terminal pipeline signal means something — this is the D-09 foundation that makes that true.
Output: `Hook::Merge` wired first in the terminal batch, an idempotent merge helper, a merged/no-op event distinction, and a cleaned CHANGELOG.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DevFlow terminal hook path → primary checkout git state | Merge/version/tag mutate real, shared repository refs; a wrong-checkout or missing merge silently corrupts release history |
| events.jsonl payload → observers (status, Hermes, 16h) | A hook that reports success without effect misleads every downstream consumer |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-16-01 | Spoofing (false success) | hooks_after_ship / run_checkout_hooks reporting ok=true with no merge | high | mitigate | Wire Hook::Merge first; emit truthful merge_result{merged:bool} (Task 1); prove effect end-to-end (Task 2) |
| T-16-02 | Tampering | VersionBump tagging the wrong checkout / pre-merge tree | high | mitigate | Merge ordered before VersionBump so compute_version runs on post-merge develop; regression test asserts tag == post-merge count (Task 2) |
| T-16-03 | Tampering / DoS | Non-idempotent re-merge double-merging an already-shipped phase | medium | mitigate | is_merged_into_develop guard makes re-run a no-op returning merged=false (Task 1) |
| T-16-04 | Repudiation | Corrupted CHANGELOG/tag history obscuring what actually shipped | low | mitigate | Remove bogus auto-entries; verify tag sequence clean (Task 3) |
| T-16-05 | Elevation / injection | git args built from untrusted input in the new merge helper | low | mitigate | Structured `Command` args only (branch name from phase u32), never shell string interpolation — matches existing git.rs idiom |
| T-16-SC | Tampering | package installs | n/a | accept | No package-manager installs in this plan (see 16-02 for the only new dependency and its blocking legitimacy checkpoint) |
