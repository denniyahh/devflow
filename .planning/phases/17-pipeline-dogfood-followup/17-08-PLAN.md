# 17-08: Preflight gate resolution spawns the agent twice

## Why

`17-REVIEW.md` CR-01, still open at `ff87f7f`, and the reason `17-VALIDATION.md`
row 6 is `⚠️ PARTIAL` and `nyquist_compliant` is `false`. This is the phase's
last blocking gap.

`run_preflight` (`main.rs:788-816`) is called from the *middle* of
`launch_stage`'s body (`main.rs:1087`), with three more launch steps still to
run after it returns:

```rust
run_preflight(&project_root, state, adapter.as_ref())?;   // line 1087
enforce_build_staleness(&project_root, state, ...)?;      // still to run
agent_result::archive_phase_files(...)?;                  // still to run
let pid = monitor::spawn_monitor(state, program, &args, ...)?;  // still to run
```

But when a check fails, `run_preflight` resolves the gate by recursing into a
**full** `launch_stage(state, None, None)` — which runs those same three steps
to completion and spawns the agent. It then returns that `Ok(())` to the outer
frame, where `?` cannot distinguish "preflight passed, carry on" from "the
retry already launched everything". So the outer frame carries on and spawns a
**second** competing agent for the same stage: a second `stage_launched` event,
and a second `archive_phase_files`/`spawn_monitor` racing the first agent's
live stdout capture and worktree.

The realistic trigger is the Ship-stage `gh auth status` check: an operator
sees the preflight gate, runs `gh auth login`, approves — and DevFlow launches
Ship's agent twice concurrently in the same worktree. It reaches every
`launch_stage` call site (`start`, `resume`, `transition`, `loop_back_to_code`,
and `handle_stage_failure`'s retry arms).

Both existing tests pre-write `{"approved":false,"note":"abort: …"}`, so only
the `Abort` arm is ever executed — the two arms that carry the bug have never
been run by a test.

Note: `handle_stage_failure`'s analogous recursive `launch_stage` calls are
*not* affected — it is only ever invoked from the top-level `advance()`
dispatcher, so there is no remaining caller work after it returns.
`run_preflight` is the only recursive-into-`launch_stage` helper that is itself
called from within `launch_stage`.

## Tasks

### Task 1: Cover both uncovered gate arms (RED)

Write both regression tests first and confirm they FAIL against current
`main.rs` — a double `stage_launched` is exactly what they must catch.

The trap: a preflight check that fails unconditionally recurses into a second
gate (the first gate's response was consumed by `Gates::cleanup`), so the test
would block on `poll_response` instead of asserting. Use a test-only adapter
whose `preflight` fails **only on first call** — an `AtomicBool`/`Cell` flag —
so the recursive `launch_stage` passes preflight, proceeds, and spawns once.
Model it on the existing `AlwaysRejectAdapter` (`main.rs:4531`), whose
`exec_command` returns `("true", Vec::new())` so `ensure_agent_binary` and
`spawn_monitor` both succeed in a tempdir.

- `run_preflight_advance_gate_launches_agent_exactly_once` — pre-seed
  `{"approved":true,...}` (→ `GateAction::Advance`), drive the full
  `launch_stage`, assert the phase event log contains exactly **one**
  `stage_launched` event.
- `run_preflight_loopback_gate_launches_agent_exactly_once` — pre-seed
  `{"approved":false,"note":"retry"}`. Per `GateAction::from_response`
  (`gates.rs:69-78`) a non-abort rejection yields `LoopBack(Stage::Code)`.
  Same single-`stage_launched` assertion.

Count across the whole event log for the phase, not just the last event —
`last_event_for_phase` cannot distinguish one launch from two.

**Verify:** both new tests FAIL against unmodified `main.rs`, each reporting 2
observed `stage_launched` events where 1 was expected. A test that passes here
is not testing the bug — fix the test before continuing.

### Task 2: Make the return type carry the continue/stop decision (GREEN)

Change `run_preflight` to `Result<bool, CliError>`, documenting the contract on
the function:

```rust
/// Returns `Ok(true)` when the caller should continue the rest of
/// `launch_stage` (preflight passed). Returns `Ok(false)` when a failing
/// check was resolved via a gate that ALREADY completed a full retried
/// launch (Advance/LoopBack) or aborted (Abort) — the caller must not run
/// any more launch steps for this invocation.
```

Replace the `return match ...` with a plain `match` whose arms `?`-propagate
(`launch_stage(state, None, None)?;`, `abort(project_root, state, &reason)?;`),
then `return Ok(false);` after it. The success path returns `Ok(true)`.

At the call site (`main.rs:1087`):

```rust
if !run_preflight(&project_root, state, adapter.as_ref())? {
    return Ok(());
}
```

Update the two existing tests for the new signature — assert the returned value
is `false` rather than discarding it, so the Abort arm's contract is pinned too.

`run_preflight_adapter_hook_override_fires`, plus the full workspace suite stay
green. `cargo clippy --workspace --all-targets -- -D warnings` clean;
