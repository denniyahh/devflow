# 17-07: Staleness gate misclassifies a descendant build as stale

## Why

Found live, not by review: Phase 17's own Validate stage was hard-blocked by the
staleness gate this phase introduced. The devflow binary was built from
`feature/phase-17` (23 commits ahead of `develop`), and the primary checkout's
HEAD was `develop`.

`embedded_commit_is_stale` runs:

```
git merge-base --is-ancestor <embedded> HEAD
```

and maps exit 1 — "embedded is NOT an ancestor of HEAD" — straight to
`Staleness::Stale`. But exit 1 covers two opposite situations:

1. embedded is older/divergent — genuinely stale
2. embedded is a **descendant** of HEAD — strictly *newer* than the checkout

Empirically confirmed against the live repo (embedded `8725304`, HEAD
`a2c314f`): forward check exits 1, reverse check `--is-ancestor HEAD <embedded>`
exits 0, proving descendant. The gate blocked a binary that was newer than the
source it was driving — the exact inverse of the "committed, forgot to rebuild"
incident it exists to catch.

Phase 17 verification scored 12/12 without catching this because both existing
ancestry tests (`wr01_clean_tree_strict_ancestor_build_is_stale_and_hard_blocks`
and the Fresh case) exercise only the ancestor direction.

## Tasks

### Task 1: Add the Ahead variant and reverse-ancestry check (RED → GREEN)

Write the failing regression test first, then implement.

Test `ahead_build_from_descendant_commit_warns_instead_of_blocking`, modelled on
the existing WR-01 fixture: two commits on a clean tree, embedded commit = the
LATER commit, then `git reset --hard` HEAD back to the earlier commit so the
embedded commit is a descendant of HEAD. Assert:

- `embedded_commit_is_stale(root, &embedded) == Staleness::Ahead`
- `staleness_outcome(true, Staleness::Ahead) == StalenessOutcome::Warn`
- `enforce_build_staleness(...)` returns `Ok(())` — no hard block for a
  self-dogfood workspace

Implementation in `embedded_commit_is_stale`: replace the bare
`Ok(Some(1)) => Staleness::Stale` arm with a reverse-ancestry probe — when
`merge-base --is-ancestor HEAD <embedded>` succeeds, the embedded commit is a
descendant, so return `Staleness::Ahead`; otherwise it is divergent or older and
stays `Staleness::Stale`. Add `Ahead` to the `Staleness` enum and a
`(_, Staleness::Ahead) => StalenessOutcome::Warn` arm to `staleness_outcome`.

`combined_staleness` needs no new branch: its early return only fires on
`Stale`, so an `Ahead` ancestry result still falls through to the mtime arm and
is correctly upgraded to `Stale` on a dirty tree with newer tracked source.

**Verify:** `cargo test -p devflow` (the new test plus every existing staleness
test, notably the WR-01 block case, must pass); `cargo clippy -p devflow --
-D warnings`; `cargo fmt --check`.

### Task 2: Confirm the fix self-permits

Rebuild the binary from this branch and confirm it no longer blocks itself when
driving the DevFlow workspace from a checkout whose HEAD is behind the build:
the staleness gate must emit a warning and proceed rather than returning the
`self-dogfood stale build blocked` error.

**Verify:** rebuilt binary launches a stage against the primary checkout without
the block; a warning is surfaced (never silent).
