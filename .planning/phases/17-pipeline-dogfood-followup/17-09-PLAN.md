# 17-09: Bound the concurrent-ship gate poll so the suite cannot wedge

## Why

`17-VALIDATION.md` GAP-2, re-measured on 2026-07-19: **2 of 5 runs (~40 %) hang
indefinitely**, and the passing runs complete in 1–2 s, so the hang is bimodal
rather than slow-path variance. With no `timeout` wrapper — which is how CI runs
it — a hung run stalls the job instead of failing it.

Confirmed pre-existing: the test was introduced by Phase 14 (`0ef85ee`) and is
present on `develop`, so this is not a Phase 17 regression. It is being fixed
here because it blocks Phase 17's Validate stage from ever returning
`verdict: pass`, and because a ~40 % chance of wedging CI is a landmine on
`develop` regardless of which phase owns it.

Observed live **twice**, independently: once during this phase's own verification
work (an 83-minute wedge, thread parked in `hrtimer_nanosleep`) and once during
plan 17-08's execution. Neither reproduced on retry, which is the point.

## Diagnosis

`concurrent_ship_advances_finish_both_phases_independently`
(`crates/devflow-cli/src/main.rs:3762`) sets up phases 31 and 32, pre-writing
**exactly one** Ship gate response each, then advances both concurrently in a
`thread::scope`.

Both phases compute the same next version and race to create tag `v2.0.1`. The
loser's ship fails, which **reopens** its `32-ship` gate — but that phase's
single pre-written response was already consumed by the first resolution. The
reopened gate therefore polls against `DEVFLOW_GATE_TIMEOUT_SECS`, which
defaults to **7 days** (`main.rs:27-38`), so the thread never returns and
`handle.join()` blocks forever.

Note there are two distinct layers here, and this plan addresses only the first:

1. **Test-level (in scope):** the test can wedge the suite indefinitely.
2. **Product-level (NOT in scope):** two phases shipping concurrently genuinely
   contend for one version tag. Whether that should serialize, retry with a
   bumped version, or fail fast with a clear error is a design question. Record
   it; do not decide it here.

## Tasks

### Task 1: Make the test incapable of wedging (RED → GREEN)

The binding constraint is **"never hangs"**, not "always passes". Establish RED
by demonstrating the current wedge — run the test in isolation under an external
timeout in a loop until a hang is observed (the re-audit needed 5 runs; budget
at least 10) and record the exit code.

Then bound it. Choose based on what the product actually guarantees, and state
the evidence for the choice in the SUMMARY:

- If concurrent same-version ships are **expected to contend**, the honest fix
  is to assert the loser's documented behavior rather than pretending both
  succeed — and to give the reopened gate a bounded response or a short
  `DEVFLOW_GATE_TIMEOUT_SECS` so the assertion is reachable.
- If they are **expected to both succeed**, the test setup is wrong: give each
  phase its own version lineage so no tag collision occurs.

If a short `DEVFLOW_GATE_TIMEOUT_SECS` is used, set it under the existing
`ENV_MUTEX` guard using the established PATH-neutralization idiom in this file
(fresh empty tempdir, not an empty string) — process-global env mutated outside
that lock will race other tests.

Do **not** change the 7-day production default. A real gate waiting on a human
should still wait.

**Verify:** run the test in isolation **at least 10 consecutive times** under a
120 s external timeout; every run must exit non-124 with the same verdict. A
single green run is explicitly not acceptable evidence — the auditor's standing
warning is that a green run of this test is "a lucky pass, not evidence." Then
`cargo test --workspace`, `cargo clippy --workspace --all-targets -- -D warnings`,
`cargo fmt --check`.
