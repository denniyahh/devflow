## Task 1: GAP-6 — `replace_version_in_contents` drops the rest of the line (RED → GREEN)

**Impl:** `crates/devflow-core/src/version.rs`, `replace_version_in_contents` (~253–297).

The matched line is reassembled as `left.trim_end() + separator + quoted_version + '\n'`.
Everything in `value` after the version token is discarded. For JSON that silently
eats the mandatory trailing comma:

```
in : {"name": "x", "version": "0.1.0", "private": true}   (version not last)
out:   "version": "2.3.4"        <- comma gone; file no longer parses
```

For TOML it eats any trailing comment (`version = "0.1.0"  # pinned`).

This is reachable in product: `Hook::VersionBump` → `version::write_version` on any
Node project where `version` is not the last key — essentially every real
`package.json`. `version_bump` then `commit_path`s the corrupted file and tags it,
landing the corruption in history.

**Establish RED first.** Add two tests to `version.rs`'s test module:

1. `write_version_preserves_trailing_comma_in_package_json` — fixture is a
   multi-key `package.json` with a key *after* `version`. After `write_version`,
   assert `serde_json::from_str::<serde_json::Value>(&contents)` is `Ok`, that the
   sibling keys still exist with their original values, and that the version reads
   back as the new one. Parsing is the assertion that matters — a substring check
   would be another vacuous fixture.
2. `write_version_preserves_trailing_comment_in_toml` — `version = "0.1.0"  # pinned`
   survives with its comment.

Both must fail against the current implementation before the fix lands. Record the
observed RED failure output in the SUMMARY.

**Then fix.** After locating the version token inside `value`, capture the remainder
and re-emit it:

- If the value is quoted, the token ends at the closing quote; the remainder is
  everything after it.
- If unquoted, the token ends at the first whitespace, `,`, or `#`.
- Emit `left.trim_end() + separator + quoted_new_version + remainder.trim_end() + '\n'`,
  so `"0.1.0",` → `"2.3.4",` and `"0.1.0"  # pinned` → `"2.3.4"  # pinned`, while a
  bare `"0.1.0"` still emits no trailing whitespace.

Keep the existing `[section]`/`split_field` handling, the `{` guard, the quote-style
detection, and the `changed` short-circuit exactly as they are.

**Verify:** `cargo test -p devflow-core version::` — and read the `N passed` line,
do not infer success from exit 0 (a filtered `cargo test` that matches nothing exits
0 and reports `0 passed`; that is not a green).

## Task 2: GAP-7 — the after-ship batch desyncs tag from changelog with no version file (RED → GREEN)

**Impl:** `crates/devflow-core/src/hooks.rs` — `version_bump` (~219–241) and
`changelog_append` (~190–217); dispatch site `crates/devflow-cli/src/main.rs:1743–1750`.

With no `Cargo.toml`/`pyproject.toml`/`package.json`, `version_bump` takes the
`else` branch (`warn!("no supported version file; tagging only")`) and **still tags**
`v{compute_version()}`. `changelog_append` runs next, calls `version::read_version`,
which errors, and falls back to the literal `"unreleased"`:

```
TAG       = v0.0.3
CHANGELOG = ## unreleased — 2026-07-19
```

That is precisely the three-way agreement row 12 exists to assert. The batch test
cannot catch it because `init_repo` (`hooks.rs:283`) unconditionally writes
`Cargo.toml`, so the `else` branch is unreachable from the test suite.

**Establish RED first.** Add
`after_ship_batch_with_no_version_file_keeps_tag_and_changelog_in_sync`:

- Split the fixture so the no-version-file case is reachable — e.g. an
  `init_repo_with_options(root, write_version_file: bool)` that `init_repo` delegates
  to with `true`. Every existing test keeps its current behavior byte for byte; do
  not edit `init_repo`'s observable effect.
- Mirror the existing batch test's `.gitignore` / feature-branch setup so the run is
  comparable, then drive `for hook in hooks_after_ship()`.
- Assert exactly one tag exists, that the CHANGELOG heading version equals the tag
  with its leading `v` stripped, and explicitly that the heading is **not**
  `unreleased`.

It must fail against the current implementation. Record the RED output in the SUMMARY.

**Then fix** by threading the tagged version instead of re-deriving it:

- Add `pub shipped_version: Option<String>` to `HookContext` (default `None`).
- Change `Hook::run` to take `&mut HookContext`.
- `version_bump` sets `ctx.shipped_version = Some(version.to_string())` after
  `git.tag(&tag)` succeeds — on both the version-file and no-version-file branches,
  since both tag.
- `changelog_append` prefers `ctx.shipped_version.clone()`, then falls back to
  `version::read_version`, then to `"unreleased"`. Keep the existing comment
  explaining why `compute_version` must not be called here (its minor derives from
  the live tag count, which `version_bump` just incremented).
- In `main.rs::run_checkout_hooks`, hoist the `HookContext` construction above the
  `for hook in batch` loop, bind it `mut`, and call `hook.run(&mut ctx)`. Its fields
  are already loop-invariant, so this removes a redundant per-iteration clone.
  **Do not change the `terminal_batch && outcome.is_err()` break** — the fail-fast
  that stops `BranchCleanup` from deleting evidence must survive this refactor.
- Update the in-crate test call sites for the `&mut` signature.

**Verify:** the new test plus `cargo test --workspace`,
`cargo clippy --workspace --all-targets -- -D warnings`, `cargo fmt --check`.
Report the actual workspace pass/fail counts in the SUMMARY — the phase baseline is
376 passed / 0 failed, so the new total should be 376 + the tests added here.
