# 17-12: WR-04 — the changelog heading is written before the tag that would make it true

## Why

The pipeline is in a **diverging** loop, observed live across two Ship attempts:

| Ship attempt | Version claimed | Reality |
|---|---|---|
| Round 1 | `1.4.26` | no tag; `Cargo.toml` at `1.3.69` |
| Round 2 | `1.4.88` | no tag; `Cargo.toml` at `1.3.69` |

Each Validate→Ship transition fires `ChangelogAppend`, which writes a dated
release heading for a version that does not exist. Ship's review correctly flags
the false external-state claim as Critical and loops back to Code. Code removes
the heading, Validate passes, the transition fires the hook **again** — and the
number is *higher* each time, because the loop's own commits feed
`commits_since_last_minor_tag()`. The pipeline cannot converge; every pass makes
the claim more wrong.

Root cause is **WR-04**, named in the Round 1 review and left unfixed:
`changelog_append` runs at the Validate→Ship transition while `version_bump`
runs in the after-ship batch. The heading is therefore written before — and
independently of — the tag that would make it true, and each hook calls
`compute_version()` separately.

Note 17-10's targeting fix is working and is not implicated: the entry now lands
in the worktree with `develop` clean. The *targeting* was fixed; the *ordering*
is what remains.

## The trap this plan must avoid

Moving `ChangelogAppend` after `VersionBump` is necessary but **not sufficient**.
`version::compute_version()` (`version.rs:142`) derives:

```rust
let minor = count_git_tags(project_root)?;
let patch = commits_since_last_minor_tag(project_root)?;
```

`VersionBump` creates a tag. So a `compute_version()` call made *after* it sees a
tag count one higher and returns a **different, larger version than the tag just
cut**. Naively reordering the hooks would replace one false heading with another,
off by one — the same defect wearing a different number.

`changelog_append` (`hooks.rs:174`) currently calls `compute_version()` directly.
In its new position it must not. There is no read-current-version helper today —
`version.rs` exposes `detect_version_file` and `write_version`, but no reader.

## Tasks

### Task 1: Add a version reader (RED → GREEN)

Add `version::read_version(project_root) -> Result<Version, VersionError>` that
parses the version out of whatever file `detect_version_file` resolves, mirroring
`write_version`'s format handling (including `[workspace.package]`, which
`write_version_replaces_in_workspace_cargo_toml` already proves is supported).

Test first: write a known version with `write_version`, read it back, assert
round-trip equality across the same file formats `write_version` supports.

**Verify:** `cargo test -p devflow-core version::`.

### Task 2: Reorder the hooks and re-source the version (RED → GREEN)

Establish RED with a test that drives the full after-ship batch and asserts the
changelog heading matches the created tag. It must fail against current `main`
(today the heading is written in the wrong batch entirely).

Then:

- `hooks_for_transition(Validate, Ship)` → `vec![Hook::DocsUpdate]`
- `hooks_after_ship()` → `Merge`, `VersionBump`, `ChangelogAppend`,
  `BranchCleanup` — `ChangelogAppend` strictly after `VersionBump`, and before
  `BranchCleanup` so a changelog failure still stops short of deleting the branch.
- `changelog_append` uses `version::read_version` instead of `compute_version`.

Preserve the terminal batch's fail-fast ordering: `run_checkout_hooks` breaks out
of the loop on the first terminal-batch error, so a failed `Merge` or
`VersionBump` must stop before `ChangelogAppend` runs. Do not weaken that.

**Verify:** the new test asserts three-way agreement — changelog heading version
== `git tag` created == version file contents. Plus `cargo test --workspace`,
`cargo clippy --workspace --all-targets -- -D warnings`, `cargo fmt --check`.

### Task 2b: Commit the entry (Round 2 WR-04)

Ordering alone does not fix this. Round 2's review found a second defect with the
same number (the review's WR-numbering collides across rounds):
`changelog_append` calls `std::fs::write` and **never commits**. `docs_update` is
the only hook that runs `git.commit_all(...)`, and it runs *first*. So the
changelog write is left dirty; `Merge` then merges the feature branch without it
and `BranchCleanup` deletes that branch, orphaning the only copy.

The review is explicit that `ae744ed` (17-10) relocated the write into the
worktree but left this root cause in place — and thereby moved it off the primary
checkout, where `git status` would have surfaced it. Retargeting improved
correctness and reduced visibility; both halves need to be true.

`changelog_append` must commit its own write, mirroring `docs_update`'s
`git.commit_all(...)` idiom, scoped to the changelog path.

**Accepted tradeoff, to be stated in the SUMMARY rather than left implicit:** with
`ChangelogAppend` ordered after `VersionBump`, the changelog commit lands *after*
the tag, so the tagged commit does not contain its own changelog entry. Making the
tag include it would require splitting `VersionBump` into write-then-tag with
`ChangelogAppend` in between — a larger restructure deliberately not attempted
here. The entry being committed and truthful is the property that matters; tag
inclusion is cosmetic and is disclosed, not hidden.

**Verify:** after the full after-ship batch, `git status --porcelain` is empty and
`git log -1 --name-only` shows `CHANGELOG.md` in a commit.

### Task 3: Record the disposition

Mark WR-04 resolved in `17-REVIEW.md`, and note in the SUMMARY that
`ChangelogAppend` deliberately moved to the terminal batch — so it now writes to
`project_root` (the release record belongs with the tag on the base branch),
while `DocsUpdate` stays in the transition batch targeting the worktree. A future
reader should not "restore" 17-10's worktree targeting to this hook.

Also record that the generated entry body is still the placeholder
"Released phase via DevFlow", which describes nothing about the phase — a
content-quality defect deliberately left out of scope here.

**Verify:** no open Critical or WR-04 entry remains in `17-REVIEW.md`.
