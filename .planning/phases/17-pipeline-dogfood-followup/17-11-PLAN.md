# 17-11: Fix CR-02 — build provenance the trigger set cannot honor

## Why

`17-REVIEW.md` **CR-02**, reclassified manual-only because both fixes it proposed
reverse a recorded design decision (`build.rs:32-35`, D-20 / review consensus #7).
The design call has now been made by the operator; this plan implements it.

The defect, reproduced by the reviewer in a fresh clone:

```
=== BUILD 1 ===
DEVFLOW_BUILD_COMMIT=6995b6f…   DEVFLOW_BUILD_DIRTY=false   DEVFLOW_BUILD_TIMESTAMP=1784493764
# edit crates/devflow-cli/src/main.rs; rebuild (the crate DOES recompile)
=== BUILD 2 ===
DEVFLOW_BUILD_COMMIT=6995b6f…   DEVFLOW_BUILD_DIRTY=false   DEVFLOW_BUILD_TIMESTAMP=1784493764
```

`build.rs` reads `git status --porcelain` (the whole working tree, `:48`) and
`SystemTime::now()` (`:67`), but declares `rerun-if-changed` only for
`HEAD`/`refs`/`packed-refs`. Neither of those two inputs can be fingerprinted by
path, so both go stale. The staleness gate — 17d's headline deliverable, whose
whole purpose is catching "you forgot to rebuild" — then certifies a binary
carrying uncommitted changes as a clean build.

**It is masked on the developer machine by accident:** `.git/packed-refs` does
not exist locally, and cargo treats a missing `rerun-if-changed` path as *always
rerun*. Any `git gc` or CI checkout (`actions/checkout@v4` produces packed refs)
exposes it. That is why no test caught it and why this phase's own dogfood runs
never surfaced it.

## The decision

Chosen: **always re-run build.rs, and stop embedding the timestamp**, keeping
commit + dirty.

The reviewer noted that dropping the caching intent recompiles `devflow-cli` on
every build. Their own analysis attributes that specifically to the timestamp —
it changes every run, so the `rustc-env` value changes, so the crate rebuilds.
Removing the timestamp removes that cause: the build script itself re-runs every
build (a few cheap git calls), but `rustc-env` only changes when the commit or
the dirty flag actually changes, so no spurious recompiles.

Retiring the timestamp also retires the mtime arm, which is where 17-10's
CHANGELOG false-positive lived. Two defect classes close together.

## Tasks

### Task 1: build.rs — always re-run, drop the timestamp

Replace the three `rerun-if-changed` lines with a single always-rerun sentinel
(`cargo:rerun-if-changed=` against a path that cannot exist), and delete the
`DEVFLOW_BUILD_TIMESTAMP` emission. Keep `DEVFLOW_BUILD_COMMIT` and
`DEVFLOW_BUILD_DIRTY`, and keep D-20's graceful degradation — absent git
metadata still emits empty/false rather than failing the build.

Replace the now-inaccurate comment at `:32-35` with the reasoning above, so the
next reader sees why the narrow trigger set was abandoned rather than assuming
it was an oversight.

**Verify:** in a fresh `git clone` of the branch (which has `packed-refs`, unlike
the dev checkout), build, edit a tracked `.rs` file, rebuild, and assert
`DEVFLOW_BUILD_DIRTY` flips `false → true`. This is the reviewer's exact
reproduction and it must now fail to reproduce. Extend
`crates/devflow-cli/tests/build_provenance.rs` with the two-build assertion.

### Task 2: Replace the mtime arm with a dirty-flag comparison

`enforce_build_staleness` and `combined_staleness` currently take
`build_timestamp: u64`. Change both to take `build_dirty: bool`, and replace
`tracked_source_newer_than_build` with a live check: does the working tree
*currently* have modified files matching `affects_compiled_binary` (the helper
17-10 added — reuse it, do not duplicate the predicate).

Decision table, replacing the mtime arm only. The ancestry arm is untouched:

| build was dirty | tree has modified build inputs now | result |
|---|---|---|
| false | yes | **Stale** — built clean, source changed since (the CR-02 case) |
| true | yes | **Indeterminate** — cannot distinguish "same dirt" from "more dirt" without a timestamp; warn, never block (Pitfall 4) |
| either | no | fall through to the ancestry result unchanged |

Update the ~20 call sites, `main.rs:1148` (pass
`env!("DEVFLOW_BUILD_DIRTY") == "true"`), and `main.rs:843` (drop
`build_timestamp` from the `workflow_started` payload; keep `dirty`).

Correct — do not delete — any test whose fixture encodes the timestamp
semantics, the same way 17-10 corrected `a.txt` → `src/lib.rs` rather than
weakening an assertion. In particular
`combined_staleness_mtime_arm_flags_dirty_tree_newer_than_build` needs renaming
and rewriting against the dirty-flag rule; its *intent* (a second signal can
flip an ancestry-Fresh result to Stale) survives.

**Verify:** `cargo test --workspace`; `cargo clippy --workspace --all-targets --
-D warnings`; `cargo fmt --check`. 17-06's `wr01_…hard_blocks`, 17-07's
`ahead_build_…`, and 17-10's `mtime_arm_ignores_non_build_files…` (renamed as
needed) must all still express their original guarantees.
