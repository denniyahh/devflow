# 17-10: DevFlow's own hooks dirty the tree, then its staleness gate blocks on the dirt

## Why

Found live: Phase 17's **Ship** stage was hard-blocked by
`self_dogfood_stale_blocked`, caused entirely by DevFlow itself.

Sequence, reconstructed from `events.jsonl` and the filesystem:

1. The Validate→Ship transition fired its hooks (`hook_run` DocsUpdate +
   ChangelogAppend, ts `1784492315`).
2. `ChangelogAppend` modified `CHANGELOG.md` at 16:18:35. The running binary
   was built at 14:40:54.
3. `combined_staleness` ran its mtime arm, saw a dirty tree with a tracked file
   newer than the build, and upgraded the verdict to `Stale`.
4. Self-dogfood workspace + `Stale` → `Block`. Ship never launched.

`git ls-files -m` reported exactly one file: `CHANGELOG.md`. A markdown file
cannot affect a compiled binary, and the pipeline had dirtied it itself seconds
earlier.

This is a *different* arm from the one 17-07 fixed. 17-07 corrected the
**ancestry** arm (descendant builds misread as stale). This is the **mtime**
arm, which `combined_staleness` applies on top of it:

```rust
let ancestry = embedded_commit_is_stale(...);   // 17-07: correctly Ahead
if ancestry == Staleness::Stale { return Stale; }
match tracked_source_newer_than_build(...) {
    Some(true) => Staleness::Stale,             // CHANGELOG.md trips this
    _ => ancestry,
}
```

## Second, worse bug found in the same incident

`ChangelogAppend` wrote to **`develop`'s** `CHANGELOG.md` — the primary
checkout — not to the worktree holding all of Phase 17's commits. The release
note for the phase is stranded on the base branch, uncommitted and divorced
from the work it describes; the branch about to merge carries no entry at all.

Cause: `run_checkout_hooks` (`main.rs:1585`) builds every `HookContext` with
`project_root: project_root.to_path_buf()`, regardless of `state.worktree_path`.

The batches split cleanly, so the fix is bounded:

- `hooks_for_transition(Validate, Ship)` → `[DocsUpdate, ChangelogAppend]` —
  content authored *about the branch*, belongs in the worktree.
- `hooks_after_ship()` → `[Merge, VersionBump, BranchCleanup]` — merging into
  the base branch, tagging it, and deleting the feature branch are operations
  on the primary checkout. **Retargeting these would be a correctness
  regression.**

## Tasks

### Task 1: Scope the mtime arm to build-affecting paths (RED → GREEN)

Write the failing test first: a self-dogfood fixture whose only modified tracked
file is `CHANGELOG.md`, with an mtime newer than `build_timestamp`, must yield
`Fresh`/`Ahead` — not `Stale` — from `combined_staleness`, and must not block.
Assert the converse in the same or a sibling test: a modified `.rs` file newer
than the build still yields `Stale` and still blocks, so the gate keeps catching
the real "committed, forgot to rebuild" case.

Implementation: filter the `git ls-files -m` results in
`tracked_source_newer_than_build` to build-affecting paths only — `*.rs`,
`Cargo.toml`, `Cargo.lock`, `build.rs`, `rust-toolchain.toml`. Everything else
(`*.md`, `.planning/`, docs) cannot change the binary and must not contribute.

Keep the filter list in one named constant with a comment explaining the
criterion ("can this file change the compiled binary?"), so extending it later
is obvious.

**Verify:** `cargo test --workspace`; `cargo clippy --workspace --all-targets --
-D warnings`; `cargo fmt --check`. The 17-06 strict-ancestor block test and the
17-07 Ahead test must both still pass.

### Task 2: Point content hooks at the worktree (RED → GREEN)

Write the failing test first: for a phase whose `state.worktree_path` is set,
the `HookContext` handed to a `hooks_for_transition(Validate, Ship)` hook must
carry the worktree path, while the context for `hooks_after_ship()` hooks must
still carry `project_root`.

Implementation: in `run_checkout_hooks`, resolve the context root per batch —
worktree for the transition batch when `state.worktree_path` is present and
exists on disk, `project_root` for the terminal batch. Fall back to
`project_root` whenever no worktree is configured, so `--no-worktree` runs are
unaffected.

Leave the checkout lock keyed on `project_root` as it is: it serializes
primary-checkout contention between phases and that property is unrelated to
where a content hook writes.

**Verify:** as Task 1, plus confirm no existing hook test regressed.
