<objective>
20b — `phase7_cli.rs` git fixtures are unreliable under CI, and the underlying
`cleanup --force` worktree-removal race is product-reachable.

Two distinct CI flakes surfaced on release-path PRs in one day: a
`git worktree remove` "Directory not empty" race
(`reference_and_cleanup_worktree_cli_flow`, phase7_cli.rs:534) and a git
object-store "invalid object" corruption mid-way through a 60-commit loop
(`start_worktree_mode_ignores_main_checkout_divergence`, phase7_cli.rs:236).

Research CONFIRMED instance 1 is a real product defect (resolving D-04):
`commands::cleanup` (commands.rs:292-335) removes every worktree
unconditionally — it never loads `workflow::list_states`, never checks
`monitor_pid`, never calls the existing `liveness()` predicate built in 18b for
exactly this. A real `devflow cleanup --force` run against a phase whose monitor
is still alive races the agent process's writes inside the worktree against
`git worktree remove`'s clean-directory scan.

Purpose: fix the product defect (liveness-gated removal + bounded retry) so the
fixture goes green as a consequence, and harden the fixtures' durability so
instance 2 stops flaking. Instance 2 is fixed FIXTURE-SIDE ONLY per D-08; the
`devflow parallel` object-store analog is deferred to backlog (see below).

Output: a liveness guard + bounded-backoff retry in `cleanup`; fixture
durability knobs in `phase7_cli.rs`; regression + idempotency tests.
</objective>
