<objective>
20d — Release-cut automation (`devflow release --check`), read-only preflight.

DevFlow automates the phase pipeline but the release-cut step
(version-bump PR → merge to `main` → tag → sync `develop` → publish) is a manual
checklist. Cutting v1.5.0 hit four distinct failures: `devflow-core` version-pin
drift, `main`/`develop` divergence (11 conflicts on the next PR), undocumented
crates.io publish ordering, and opaque tag-signing (`ssh_askpass`) failure.

Purpose: a read-only preflight that converts the four prose checklist steps into
structured checks run before attempting the tag. **Ceiling is `--check` ONLY
(D-03):** no executor that runs the merge/tag/sync/publish sequence — that is a
deferred backlog item (see below). This plan MUST NOT drift toward the executor.

**Blocks on 20a (20-01):** the self-pin check asserts 20a's fixed invariant. It
must verify the general rule (`every local-path self-pin == workspace version`),
NOT encode today's hand-patched value as the expected state.

Research (Pattern 4) verified live that THIS repo uses `gpg.format=ssh`, so the
signing check must be `gpg.format`-aware from the start — `ssh-add -l`'s three
exit codes (2=no agent, 1=agent-but-empty, 0=keys listed) map to three distinct
actionable messages. A GPG-only check would miss the failure this project's own
release actually hit.

Output: a `devflow release --check` command with four `Check`-shaped, read-only
checks, mirroring `doctor`'s structure; a new `release_check.rs` test file;
regenerated help snapshot + OPERATIONS.md.
</objective>
