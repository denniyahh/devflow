<objective>
20e â€” Manual ship override (`devflow ship --phase N [--force]`).

`devflow gate approve` assumes a healthy pipeline: `respond()` refuses when no
gate is open (`gates.rs:186` `NoOpenGate`), and approving only WRITES a response
file â€” a live monitor polling that path is what actually advances the workflow.
If the monitor is dead, the approval sits unconsumed forever. This is the
recovery gap when that assumption fails, which on this project's dogfood history
is not rare.

Research (Pattern 5) independently re-derived and CONFIRMED CONTEXT.md's
D-01/D-02 design correct against source: keep ONE on-disk gate-response record,
add a SECOND out-of-process consumer. `devflow ship --phase N` reads the
already-written Ship response directly and, on `GateAction::Advance`, calls the
same `pipeline_gate::finish_workflow` the live poll loop would have called
(`pipeline_gate.rs:130`) â€” `finish_workflow` already owns the fail-closed
retry-gate-reopen and terminal `workflow_finished` emission; nothing about it
needs reimplementing for a second caller.

`VersionBump`, so it inherits 20a's self-pin fix for free â€” another reason 20a
ships first. Also depends on 20-04 (shares `main.rs`/CLI dispatch surface with
20d; sequenced to keep the conflict surface in one review).
