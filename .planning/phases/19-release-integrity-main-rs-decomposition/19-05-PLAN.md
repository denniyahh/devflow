<objective>
Verify 19g actually works, by dogfooding the contract on itself.

`19-VALIDATION.md` § "Manual-Only Verifications" records why this cannot be
automated: the subject under test is an LLM review pass, not a deterministic
function. No assertion can prove judgment quality. The test is therefore a
blocking human checkpoint: construct a diff that the contract should reject,
run the review, and confirm both that it is rejected and that the rejection
cites the contract.

Purpose: `19-RESEARCH.md` rates the 19g wiring mechanism MEDIUM confidence —
the reviewer agent's own instructions say it applies project-skill rules, but
this repository has never exercised that path before. This checkpoint converts
that from an assumption into an observation.
Output: recorded review output for two non-compliant diffs (two distinct
anti-pattern shapes), a compliant control, and two edge cases.

    Plan 19-04 created `.claude/skills/ai-change-acceptance/` — this
    repository's first project skill — encoding the five D-19 acceptance
    requirements and four rejection patterns, plus an `## AI Change Acceptance`
    section in `CONTRIBUTING.md`.

    Before this checkpoint runs, the executor prepares three scratch diffs in
    the working tree (uncommitted, or on a throwaway branch that is deleted
    afterwards):

    **Diff A — non-compliant, shape 1 (assert-a-constant-against-itself).** A
    new test in an existing `#[cfg(test)] mod tests` block that violates D-19
    rejection patterns 1 and 3 simultaneously: it asserts a constant against
    itself and exercises no production function. Give it a plausible name so the
    violation is in the body, not the label — the point is to test whether the
    reviewer reads the assertion, not the identifier.

    **Diff C — non-compliant, shape 2 (reproduces the production algorithm).**
    (OpenCode review: one shape only proves the contract catches one shape.) A
    new test that re-implements the production algorithm inside the test body
    and then asserts the production function equals that re-implementation — so
    the test passes as long as the two copies agree, even if both are wrong.
    This is D-19 rejection pattern 2, the `ReviewerSetTestAdapter` shape named
    in `.planning/TEST-SUITE-QA-REVIEW.md`. Pick a small pure function in
    `devflow-core` as the subject and mirror its logic in the test. This is a
    genuinely DIFFERENT anti-pattern from Diff A: Diff A tests nothing, Diff C
    tests a tautology against a duplicated algorithm.

    **Diff B — compliant control.** A small test that fails before a
    corresponding one-line production change and passes after it, asserting at a
    public boundary. Any trivially reversible behavior in `devflow-core` is a
    suitable subject.

    All three diffs are discarded after the review, whatever the outcome.
  </what-built>

       `/gsd-code-review 19 --files <the file Diff A changed>`.

       naming the skill, one of the four rejection patterns, or one of the five
       requirements. A finding that merely says the test is "low value" or
       "could be stronger" does **not** pass this checkpoint: that is generic
       style advice the reviewer would have produced without the skill, so it
       proves nothing about the wiring.
    4. Replace Diff A with **Diff C** (reproduces the production algorithm) and
       re-run `/gsd-code-review 19 --files <the file Diff C changed>`. Confirm it
       produces a finding against Diff C's test and that the rationale cites the
       contract (naming pattern 2 / the reproduce-the-algorithm shape, or the
       skill). Record it verbatim. This is the second distinct anti-pattern
       shape — proving the contract catches more than the most obvious case.
    5. Replace the non-compliant diff with Diff B and re-run
       `/gsd-code-review 19 --files <the file Diff B changed>`.
    6. Confirm Diff B is **not** flagged by the acceptance contract. A contract
       that rejects everything is not discriminating and is useless as a gate.
       Record this output verbatim too.
    7. Discard all scratch diffs. Confirm `git status --porcelain -- crates/` is
       empty and that no throwaway branch remains.

    **If step 3 or step 4 fails** — the reviewer flags the test but does not cite
    the contract, or does not flag it at all — do **not** adjust the diff to make
    it more obviously bad, and do **not** weaken the skill. Record exactly what
    the reviewer said, state which failure mode occurred and for which diff, and

    **Edge cases to confirm and record, not skip:**
    - An **empty diff**: run the review with no changes staged. Confirm the
      contract produces no finding — it must not fire on nothing.
    - A diff that changes **only production code and no test at all**: confirm
      the reviewer flags the absence of a regression test (D-19 requirement 1),
      rather than staying silent because there is no test to criticise. This is
      the boundary case — zero tests is the limit, and one step past it is the
      missing-test finding.
