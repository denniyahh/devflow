<objective>
Verify 19g actually works, by dogfooding the contract on itself.

`19-VALIDATION.md` § "Manual-Only Verifications" records why this cannot be
automated: the subject under test is an LLM review pass, not a deterministic
function. No assertion can prove judgment quality. The test is therefore a
blocking human checkpoint: construct a diff that the contract should reject,
run the review, and confirm both that it is rejected and that the rejection
cites the contract.

Purpose: `19-RESEARCH.md` rates the 19g wiring mechanism MEDIUM confidence —
the reviewer agent's own instructions say it applies project-skill rules, but
this repository has never exercised that path before. This checkpoint converts
that from an assumption into an observation.
Output: recorded review output for a non-compliant diff and a compliant control.
</objective>

  <what-built>
    Plan 19-04 created `.claude/skills/ai-change-acceptance/` — this
    repository's first project skill — encoding the five D-19 acceptance
    requirements and four rejection patterns, plus an `## AI Change Acceptance`
    section in `CONTRIBUTING.md`.

    Before this checkpoint runs, the executor prepares two scratch diffs in the
    working tree (uncommitted, or on a throwaway branch that is deleted
    afterwards):

    **Diff A — deliberately non-compliant.** A new test in an existing
    `#[cfg(test)] mod tests` block that violates at least two of the four
    rejection patterns simultaneously: it asserts a constant against itself
    (pattern 1 and pattern 3), and it exercises no production function. Give it
    a plausible name so the violation is in the body, not the label — the point
    is to test whether the reviewer reads the assertion, not the identifier.

    **Diff B — compliant control.** A small test that fails before a
    corresponding one-line production change and passes after it, asserting at a
    public boundary. Any trivially reversible behavior in `devflow-core` is a
    suitable subject.

    Both diffs are discarded after the review, whatever the outcome.
  </what-built>

  <how-to-verify>
    1. With Diff A present in the working tree, run `/gsd-code-review` scoped to
       the changed file.
    2. Read the output. Confirm it produces a finding against the new test in
       Diff A. Record the finding verbatim in the SUMMARY.
    3. Confirm the finding's rationale references the acceptance contract — by
       naming the skill, one of the four rejection patterns, or one of the five
       requirements. A finding that merely says the test is "low value" or
       "could be stronger" does **not** pass this checkpoint: that is generic
       style advice the reviewer would have produced without the skill, so it
       proves nothing about the wiring.
    4. Replace Diff A with Diff B and re-run `/gsd-code-review` on the same
       scope.
    5. Confirm Diff B is **not** flagged by the acceptance contract. A contract
       that rejects everything is not discriminating and is useless as a gate.
       Record this output verbatim too.
    6. Discard both diffs. Confirm `git status --porcelain -- crates/` is empty
       and that no throwaway branch remains.

    **If step 3 fails** — the reviewer flags the test but does not cite the
    contract, or does not flag it at all — do **not** adjust Diff A to make it
    more obviously bad, and do **not** weaken the skill. Record exactly what the
    reviewer said, state which of the two failure modes occurred, and report it.
    The correct remedy is sharper wording in
    `rules/test-signal-rejection.md` — specifically, more concrete recognition
    criteria for the pattern that was missed — which is a follow-up change to
    plan 19-04's artifacts, re-verified by re-running this checkpoint.

    **Edge cases to confirm and record, not skip:**
    - An **empty diff**: run the review with no changes staged. Confirm the
      contract produces no finding — it must not fire on nothing.
    - A diff that changes **only production code and no test at all**: confirm
      the reviewer flags the absence of a regression test (D-19 requirement 1),
      rather than staying silent because there is no test to criticise. This is
      the boundary case — zero tests is the limit, and one step past it is the
      missing-test finding.
  </how-to-verify>

  <verify>
    <human-check>The reviewer flagged Diff A citing the contract by name, did not flag Diff B, produced nothing on an empty diff, and flagged the missing regression test on a production-only diff. All four outputs recorded verbatim in the SUMMARY.</human-check>
  </verify>

  </acceptance_criteria>
