<objective>
Close 19a-WR-01: a DevFlow run writes unredacted agent stdout, gate context, and
operator-identifying paths under `.devflow/`, and a downstream user's routine
`git add . && git commit` sweeps all of it into their repository's history.

Fix per D-14 (CORRECTED): introduce a new `workflow::ensure_devflow_dir` that
creates the directory and writes a self-ignoring `.devflow/.gitignore`
containing `*`, and convert all 7 production `create_dir_all` sites to call it.
Add a regression test proving every `.devflow/`-writing path — including the
`sequentagent`/`parallel` path that never touches `save_state` — produces the
`.gitignore`.

Purpose: this is the only defect in the phase whose blast radius reaches
outside this repository.
Output: one new public function, 7 converted call sites, one new integration
test file.
</objective>

    - Given a fresh temp dir, calling `ensure_devflow_dir(root.join(".devflow"))` creates the directory and leaves `.devflow/.gitignore` whose trimmed content equals `*`.
    - Calling it a second time on the same directory succeeds and leaves the file byte-identical.
    - If `.devflow/.gitignore` already exists with different content (e.g. `# owned by something else`), the call succeeds and leaves that content untouched.
    - Given a nested path `root/.devflow/history/phase-01`, the call creates the full nested chain and writes the `.gitignore` into `root/.devflow`, not into the leaf.
    - Given a path with no `.devflow` ancestor (e.g. `root/plain/sub`), the call creates the directories and writes no `.gitignore` anywhere.
    - Two concurrent calls on the same fresh directory both return `Ok` (no `AlreadyExists` error escapes).
  </behavior>

  <action>
    Add `pub fn ensure_devflow_dir(dir: &Path) -> std::io::Result<()>` to
    `crates/devflow-core/src/workflow.rs`, placed immediately after the existing
    `devflow_dir` accessor at `:33-35` so the pure accessor and the
    side-effecting constructor sit together and the contrast is visible in one
    screen.

    Contract, stated in the `///` doc comment verbatim as the function's
    documented behavior:
    1. `std::fs::create_dir_all(dir)?` — creates `dir` and all missing parents.
    2. Scan `dir` and its ancestors, shallowest-first (nearest the filesystem
       root), for the first path component equal to `.devflow`. Shallowest-first
       is the deterministic tie-break when a path nests `.devflow` twice
       (e.g. a fixture rooted inside another `.devflow`); document that choice
       in the doc comment.
    3. If such an ancestor exists, write `<ancestor>/.gitignore` with the exact
       bytes `*` followed by a newline, using
       `std::fs::OpenOptions::new().write(true).create_new(true)` so the write
       is skipped when the file already exists and a lost race against a
       concurrent creator surfaces as `ErrorKind::AlreadyExists`, which this
       function maps to `Ok(())`. Any other I/O error propagates with `?`.
    4. If no `.devflow` ancestor exists, the function is exactly equivalent to
       `create_dir_all`.

    Return `std::io::Result<()>` — not `Result<_, WorkflowError>`. This is
    deliberate and load-bearing: the 7 conversion sites in Task 2 live in six
    different modules returning six different error enums (`WorkflowError`,
    `GateError`, `MonitorError`, `ResultError`, `ShipError`, `LockError`), and
    every one of them already carries an `Io(#[from] std::io::Error)` variant
    (verified at HEAD), so `?` converts at every site with no signature churn.
    Record that reasoning in the doc comment.

    Document why this function exists rather than folding the behavior into
    `devflow_dir`: cite D-14's rejected alternative — `devflow_dir` is a pure
    path accessor invoked from read-only commands and from tests that assert on
    the returned path, and giving it filesystem side effects is exactly the
    class of behavioral change this phase exists to avoid.

    Add the six unit tests from `<behavior>` in the existing
    `#[cfg(test)] mod tests` block at the bottom of `workflow.rs`, following the
    file's existing `tempfile::tempdir()` fixture style. For the concurrency
    case, spawn two `std::thread` handles that both call the function on the
    same fresh path and assert both return `Ok`.

    No `.unwrap()` in the production function — use `?` throughout. Every public
    item gets a `///` doc comment.
  </action>
