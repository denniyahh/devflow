<extraction_procedure>
This procedure applies to every cluster move in this plan and in 19-08 / 19-09.
Follow it literally.

1. Re-derive the cluster's current line range with
   `rg -n '^(pub\(crate\) )?(fn|struct|enum|const|impl) ' crates/devflow-cli/src/main.rs`.
   The `19-SPLIT-BASELINE.md` ranges were measured before this plan's own edits
   and before any sibling move; **never reuse a stale range**.
2. Extract with `sed -n 'START,ENDp' crates/devflow-cli/src/main.rs > /tmp/<cluster>-prod.rs`.
3. Extract each test region the same way into `/tmp/<cluster>-tests.rs`.
   Test regions are non-contiguous; extract each separately and concatenate.
4. Assemble the new file: a module doc comment, the `use` block, the production
   extract, then `#[cfg(test)] mod tests { use super::*; use crate::test_support::*; ... }`
   wrapping the test extract.
5. Delete the extracted ranges from `main.rs`, highest line number first so
   earlier ranges do not shift.
6. Add the `mod` declaration to `main.rs`.
7. Add `pub(crate)` only where the compiler demands it. Compile, read the error,
   add the keyword, repeat. Do not pre-emptively mark everything public â€” the
   compiler's error list is the precise, minimal surface, and anything beyond it
   is unjustified API.
8. Fix `use` paths the compiler reports. Prefer `use crate::X;` for items now in
   a sibling module and `use crate::CliError;` for crate-root items.
9. Commit production and tests together, in one commit.
10. Immediately run the per-function diff for this cluster (see `<verify>`), not
    at the end of the phase. A drift caught now is one file's worth of
    investigation; caught in wave 6 it is four.
</extraction_procedure>
