<objective>
Land 19g: the AI change acceptance contract.

Codex's 2026-07-21 test-suite QA pass (`.planning/TEST-SUITE-QA-REVIEW.md`,
P0 recommendation #2) found tests in this repository that assert constants,
reproduce the production algorithm inside the test, or compare a function call
with itself — tests that pass regardless of whether the code works. The
contract states, as reviewable criteria, what an AI-authored change must
demonstrate before it is accepted, and which test shapes are rejected outright.

Per D-18 it is wired into `/gsd-code-review`'s actual criteria — via this
repository's project-skill directory, which is the project-scoped extension
point the reviewer already auto-discovers — plus prose in `CONTRIBUTING.md`.
Per D-18 there is deliberately **no** new lint tooling.

Purpose: enforcement already exists at the review-before-Ship gate. This plan
supplies the criteria that gate applies.
Output: a new `.claude/skills/` directory with one skill, and a
`CONTRIBUTING.md` section.
</objective>

    Create `.claude/skills/ai-change-acceptance/SKILL.md` as a lightweight index.
    It must contain: a one-line description of what the skill governs; when it
    applies (any change authored or substantially modified by an AI agent, which
    on this project is effectively every change); a compact numbered list of the
    five acceptance requirements and the four rejection patterns as one-line
    summaries; and pointers to the two `rules/` files for detail. Target roughly
    60–130 lines. Do not inline the detailed criteria here — that is what makes
    discovery cheap.

    Create `.claude/skills/ai-change-acceptance/rules/change-acceptance.md`
    containing the five D-19 acceptance requirements, each as its own section
    with three parts: the requirement stated as an assertion about the diff;
    the check a reviewer performs to decide it; and the observable signature of
    a failure. The five, verbatim from D-19:
    1. A regression test that fails before the change.
    2. At least one assertion at a public or otherwise stable boundary.
    3. Evidence the test fails for the intended reason — not for a setup error,
       a compile failure, or an unrelated panic.
    4. Full affected-package tests, plus `cargo clippy --workspace --all-targets -- -D warnings`,
       plus `cargo fmt --check`.
    5. Independent review of both the implementation **and** the test signal —
       reviewing only the implementation is how a test that cannot fail gets
       through.

    For requirement 4, state the two project-specific false-green traps this
    repository has actually been bitten by, because a reviewer who does not know
    them will accept a green that means nothing:
    - `cargo test --exact` with a bare test name matches nothing and still exits
      0. Acceptance requires reading the reported pass count, not the exit
      status.
    - The package is `devflow`, not `devflow-cli`. A `-p devflow-cli` invocation
      does not run the CLI unit tests.
    Also record that `cargo test --lib` does not work on this binary-only crate
    (confirmed dead end, STATE.md 18-01 entry) — use the bare form.

    Create `.claude/skills/ai-change-acceptance/rules/test-signal-rejection.md`
    containing the four D-19 rejection patterns, each with: the pattern named;
    why it produces a test that cannot fail; how to recognise it in a diff; and
    a worked example. The four:
    1. Tests that only assert constants.
    2. Tests that reproduce the production algorithm inside the test body, so
       the test and the code fail together and pass together.
    3. Tests that compare a function call with itself.
    4. Tests that grep implementation text without asserting a runtime contract.

    Draw at least one worked example from `.planning/TEST-SUITE-QA-REVIEW.md`
    (the `ReviewerSetTestAdapter` case) so the rules are grounded in this
    repository's own history rather than generic advice. Where no in-repo
    example exists for a pattern, write a minimal synthetic Rust example and
    label it as synthetic.

    Add a short closing section to `test-signal-rejection.md` stating the
    boundary of the contract: a test that greps source text is not
    automatically rejected when the property under test genuinely is a source
    property (for example, an assertion that a specific argument is absent from
    a specific function body). What is rejected is grepping as a **substitute**
    for a runtime contract that could have been asserted directly.

    Every file gets a title heading and a `**Applies to:** DevFlow` line so a
    reader landing on it out of context knows its scope.
  </action>
