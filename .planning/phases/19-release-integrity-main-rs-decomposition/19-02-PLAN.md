<objective>
Close 19a-WR-02: `workflow_started_payload` (`main.rs:894-906`) writes
`std::env::current_exe()`'s **full absolute path** into `events.jsonl`. On a
typical workstation that path embeds the operator's home directory and OS
username, and `OPERATIONS.md` tells operators this file is safe to tail and
paste into a bug report or chat.

Fix per D-15: emit only `current_exe().file_name()`. The field stays, the
schema stays, the useful binary-name signal stays, and the PII goes.

Purpose: this is the second half of D-13 — WR-01 stops the file being
published, WR-02 stops the sensitive data being written in the first place.
Either alone leaves half the problem open.
Output: one changed expression plus a strengthened assertion in the existing
test.
</objective>

  <action>
    In `workflow_started_payload` (`main.rs:894`), change the `exe_path` value
    expression so that after `std::env::current_exe().ok()` it takes the final
    path component before stringifying: chain `.and_then(|p| p.file_name().map(|n| n.to_string_lossy().into_owned()))`
    in place of the current `.map(|p| p.display().to_string())`.

    Use `to_string_lossy().into_owned()` rather than `to_str()`: a non-UTF-8
    binary name must still produce a string rather than silently collapsing to
    null, because null is the field's documented "could not determine the
    executable" signal and conflating the two would make the field ambiguous.
    State that reasoning in a short `//` comment on the changed expression,
    naming the decision as the WR-02 redaction.

    Leave every other key in the `serde_json::json!` literal byte-identical —
    `agent`, `mode`, `worktree`, `version`, `commit`, `dirty`. In particular do
    not touch `worktree`, which also carries a full path: it is a
    project-relative working directory the operator chose and is not part of
    this decision's scope. Note that observation in the SUMMARY as a
    surfaced-not-absorbed finding rather than fixing it here.

    Then strengthen `workflow_started_payload_carries_build_provenance`
    (`main.rs:6867`). Keep every existing assertion. Add an assertion that when
    `exe_path` is a string it contains neither a forward slash nor a backslash,
    with a failure message naming WR-02 and the `OPERATIONS.md` tail-and-paste
    exposure so a future reader knows why the constraint exists. Add an
    assertion that the `exe_path` key is still present in the object, so a later
    refactor cannot satisfy the redaction assertion by deleting the field.

    Update the test's `///` doc comment to record that it now also pins the
    WR-02 redaction, alongside the CR-02 `build_timestamp` guard it already
    carries.
