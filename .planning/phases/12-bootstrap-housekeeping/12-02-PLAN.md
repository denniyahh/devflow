<objective>
Fix WR-06 (reliability-critical) and IN-04 (trivial idiom).

WR-06: when a coding agent reports a rate limit whose reason string lacks the `rate limited until ` prefix, `retry_after_from_reason` falls through to the raw reason via `.or(reason)`. That unparseable string reaches `cron_schedule_from_retry_after`, which falls back to `"* * * * *"` — a cron that fires every minute until a human removes it. This is a self-inflicted resource/DoS vector for an unattended tool.

IN-04: `test_cmd` invokes `cargo fmt -- --check`; the canonical form is `cargo fmt --check`.

Purpose: Guarantee an unparseable rate-limit reason degrades to a safe, clearly-labelled "no auto-resume scheduled" state rather than a runaway cron.
Output: `retry_after_from_reason` returns `"unknown"` on unparseable input; the cron builder refuses to emit an every-minute schedule for an unparseable retry time; `test_cmd` uses the idiomatic fmt flag.
</objective>
