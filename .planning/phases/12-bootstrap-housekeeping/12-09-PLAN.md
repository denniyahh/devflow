<objective>
Cover the orchestration-core paths in main.rs that 11-VALIDATION.md flagged as only exercised via manual/e2e flows (12f): `advance()` orchestration, `consecutive_failures` reaching `MAX_CONSECUTIVE_FAILURES`, and the `abort` gate path.

Testability constraint (important): `advance()`/`transition()`/`loop_back_to_code` reach `launch_stage → monitor::spawn_monitor`, which spawns a real detached process and re-invokes the devflow binary — unsafe and flaky to trigger inside a unit test. The **terminal, non-spawning** paths ARE safely testable and cover the substantive risk:
- Ship success → `handle_ship_outcome` → approved gate → `finish_workflow` (runs after-ship hooks, cleans gates, clears state) — NO spawn.
- Validate at the failure threshold → forced gate → `abort` (clears state) — NO spawn.
This plan tests those seams with pre-seeded gate responses. The spawning transitions remain covered at the decision level by existing `mode::should_gate`/`hooks_for_transition` tests; wiring a launcher seam to unit-test the spawn paths is a refactor that is OUT OF SCOPE for this housekeeping phase (note this in the SUMMARY).

Purpose: Real, non-flaky coverage of advance's terminal orchestration and the failure-threshold→gate→abort path.
Output: inline main.rs tests seeded via a temp git repo, capture files, and gate response files.
</objective>
