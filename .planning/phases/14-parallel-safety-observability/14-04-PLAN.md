# 14-04: Observability — events.jsonl, devflow logs, richer status (14c)

**Wave 4** — blocked on 14-01..03 (events carry phase identity; logs/status
read the per-phase state model).

## Tasks

1. **`events.rs` (new, devflow-core)** — append-only `.devflow/events.jsonl`.
   Schema v1, one JSON object per line:
   `{"v":1,"ts":<unix-secs>,"phase":N,"event":"<kind>", ...kind fields}`.
   Kinds: `workflow_started`, `stage_launched`, `advance_evaluated`,
   `transition`, `loop_back`, `gate_fired`, `notify_fired`, `gate_resolved`,
   `hook_run`, `workflow_finished`, `workflow_aborted`. Single `write_all`
   of a full line via `OpenOptions::append` (atomic for line-sized writes);
   emission is fail-soft (warn, never abort the workflow).
2. **Emit from the CLI orchestration points** — `start`, `launch_stage`,
   `advance`, `transition`, `loop_back_to_code`, `run_gate` (fire/notify/
   resolve), hook loops, `finish_workflow`, `abort`.
3. **`devflow logs [--follow] [--phase N] [--stderr]`** — print the phase's
   stdout (or stderr) capture file; phase defaults to the single active
   state, else the most-recently-modified capture. `--follow` polls for
   appended bytes and terminates once the exit file exists and output is
   quiescent.
4. **Richer `devflow status`** — per active phase: stage, mode, gate
   state, agent + pid liveness, elapsed (recover's age formatting), and the
   phase's last event from events.jsonl as "last action".

## Tests

- emitted lines parse as JSON with v/ts/phase/event; appends never truncate.
- logs: picks the right phase file, `--follow` terminates after exit file.
- status: two active phases both shown with last-action lines.

## Acceptance

A running loop is observable end-to-end from another terminal with `devflow
status` + `devflow logs --follow`, and any frontend can consume
`.devflow/events.jsonl` without new integration code (Phase 16 dependency).
