# 14-01: Per-phase state files + phase-threaded advance (14a core)

**Wave 1** — the state-model change everything else builds on.

## Tasks

1. **Per-phase state files** (`workflow.rs`)
   - `state_path(root, phase)` → `.devflow/state-{phase:02}.json`.
   - `load_state(root, phase)` / `clear_state(root, phase)` take the phase;
     `save_state(&state)` keys off `state.phase`.
   - `list_states(root) -> Vec<State>` enumerates `.devflow/state-*.json`
     (sorted by phase) for `status`/`recover`/`logs`.
   - One-shot legacy migration: any load/list first moves a legacy
     `.devflow/state.json` to its per-phase name.

2. **Thread the phase through the monitor** (`monitor.rs`, `main.rs`)
   - Monitor script ends with `{binary} advance {root} --phase {NN}`.
   - `Advance` gains `--phase N`; `advance()` keys the per-phase lock on the
     argument directly — the pre-lock `load_state` read is deleted (its
     TOCTOU guard becomes unnecessary by construction).
   - Fallback when `--phase` is absent (legacy in-flight monitor): migrate
     legacy state, use the single active phase if exactly one, else error.

## Tests

- state round-trip per phase; two phases' states coexist without clobbering.
- legacy `state.json` migrates on load/list and disappears.
- `advance --phase N` operates on phase N's state while phase M's is untouched.
- monitor e2e: spawned script invokes `advance` with the right `--phase`.

## Acceptance

Two `start`s for different phases persist two state files; each phase's
monitor advances only its own stage machine (13-DEFERRED-CR-03 scenario 1).
