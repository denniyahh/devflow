<objective>
Make `devflow doctor` project-aware (18a). Today `doctor()` takes `_project_root`
unused and only probes external tools and `RUST_LOG` — it can tell you `git` is
installed but not that phase 7's state says `Code` while its agent died an hour ago
and an unanswered gate is sitting on disk. Extend it to reconcile state vs. events vs.
live PIDs vs. gates vs. branch existence, and report a repair plan while mutating
nothing.

Purpose: two silent monitor deaths in the Phase 17 dogfood run cost ~4h and were
found only via `ps`. `doctor` is the command an operator reaches for; it must answer
"what is actually wrong with this project" rather than "is git installed".

Output: a pure, unit-testable reconciliation core plus its wiring into `doctor`'s
existing text and `--json` renderers.
</objective>

<threat_model>
## Trust Boundaries

This is a local-only developer CLI: no network listener, no authentication, no
multi-tenancy, and no untrusted user input. The only boundaries that carry real
risk for this plan are the ones where machine-local data becomes shareable output.

| Boundary | Description |
|----------|-------------|
| local filesystem → `doctor` stdout | `.devflow/` state, event, and gate contents are read and rendered to a terminal that operators routinely paste into issues and PRs |
| local git repo → `doctor` stdout | branch names and `git rev-parse` results cross into rendered output |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-01 | Information Disclosure | `render_reconciliation` / `PhaseFinding.detail` | medium | mitigate | Detail and repair strings carry phase numbers, stage names, and PIDs only — never a filesystem path. This is the WR-02 class (17-REVIEW.md), where `exe_path` wrote the developer's absolute home directory and OS username into `.devflow/events.jsonl`. Task 1's action forbids paths in these strings. |
| T-18-02 | Tampering | `doctor` read-only contract | high | mitigate | `doctor` calls no mutating API (`save_state`, `events::emit`, `Gates::cleanup`, `recover::clean*` are all forbidden in this plan's code). A diagnostic that silently repairs state would make the next false-green unattributable. Enforced by `doctor_is_read_only_on_a_mismatched_project`, which runs `doctor` twice and asserts `.devflow/` is unchanged. |
| T-18-03 | Denial of Service | `collect_phase_facts` git subprocess | low | accept | One `git rev-parse` per active phase on an operator-invoked command. Bounded by the active-phase count, which is operator-controlled and single-digit in practice. |
| T-18-04 | Repudiation | `doctor` emits no event | low | accept | Deliberate: `doctor` is read-only by contract (T-18-02), so it cannot record its own invocation without violating that contract. Invocation history is out of scope for a local diagnostic. |
</threat_model>
