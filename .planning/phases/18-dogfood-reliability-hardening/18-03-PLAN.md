<objective>
Close the "who watches the watcher" gap (18b). `monitor_pid` is emitted to
`events.jsonl` at launch but never persisted to `State` and never liveness-checked.
`devflow status` probes only `agent_pid`, so a dead monitor renders identically to a
healthy between-stages moment — the agent has exited and nothing has spawned yet.

Purpose: two silent monitor deaths in the Phase 17 run cost roughly four hours and
were found only by running `ps`. Persist the pid, probe it, and make
"stuck — needs `devflow resume`" a representable, distinguishable state.

Output: a persisted `State.monitor_pid`, a pure four-way `liveness()` predicate, and
its rendering in both `status` and `doctor`.
</objective>

<threat_model>
## Trust Boundaries

Local-only developer CLI. No network, no auth, no untrusted input. The boundaries
that matter here are process-identity data crossing from the OS into persisted state
and rendered output.

| Boundary | Description |
|----------|-------------|
| OS process table → persisted `State` | a monitor PID is recorded to `.devflow/state-NN.json` and later trusted as a liveness target |
| persisted `State` → `status` / `doctor` stdout | PIDs and stage names cross into output operators paste into issues |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-08 | Spoofing | `monitor_pid` liveness probe | medium | mitigate | A PID read from disk can be stale and recycled onto an unrelated process, so a liveness `true` is not proof of the ORIGINAL monitor. Accepted as a limitation of PID-based probing, bounded by reusing the already-hardened `agent::agent_running`, which rejects pid 0 (would signal the caller's own process group) and values above `i32::MAX` (would wrap negative and probe every signalable process). No second probe is written. |
| T-18-09 | Information Disclosure | `status` / `doctor` monitor row | medium | mitigate | New output lines carry phase number, stage, and PID only — no filesystem path, no username. This is the WR-02 class (17-REVIEW.md), where `exe_path` wrote the developer's absolute home directory and OS username into `.devflow/events.jsonl`. Task 2's action forbids paths in the new lines. |
| T-18-10 | Denial of Service | silent monitor death (the defect itself) | high | mitigate | This is the availability failure the requirement exists to close: a dead monitor currently renders identically to a healthy between-stages moment, so a wedged phase is invisible and burns operator hours. Mitigated by persisting the PID, probing it, and representing `Stuck` distinctly with a named recovery verb. |
| T-18-11 | Tampering | `monitor_pid: None` misclassified as `Stuck` | high | mitigate | A state file written by a pre-18b binary carries no `monitor_pid`. Reporting every such phase as stuck would produce alarm fatigue and train operators to ignore the signal. `liveness` matches the `None` arm first and returns `Unknown`; `check_dead_monitor` fires only on `Stuck`. Pinned by `liveness_unknown_when_no_monitor_recorded`, `monitor_pid_absent_from_json_defaults_to_none`, and `reconcile_is_silent_when_monitor_pid_is_unrecorded`. |
| T-18-12 | Repudiation | monitor identity only in `events.jsonl` | low | accept | After this plan the PID is in both the event log and `State`; the event log remains append-only and unsigned, which is appropriate for a local single-operator tool. |
</threat_model>
