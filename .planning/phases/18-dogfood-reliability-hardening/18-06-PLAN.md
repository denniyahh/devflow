<objective>
Evaluate build staleness against the right tree (18c). `enforce_build_staleness`
compares the binary's embedded commit to `project_root`'s HEAD. For a worktree-based
phase the code under test lives on the worktree branch, so a binary two hours behind
that branch classifies as `Ahead` — a warning — because it is still a descendant of
`develop`. This is the root cause of Round 4 CR-01: a stale binary silently ran the
pre-17-12 hook batch and re-emitted a false changelog heading.

Purpose: this is the enforcement mechanism for a standing dogfood rule. When a live bug
is fixed mid-run and the pipeline is sent back to re-validate, the binary MUST be
rebuilt before resuming. The monitor resolves its binary via `current_exe()` at spawn
and its wrapper hardcodes the primary path, so running a worktree binary directly does
not propagate, and `cargo run` from the primary checkout silently reinstalls a pre-fix
build. Until this ships the rule is manual; after it, the gate enforces it.

Output: `execution_root` threaded through the ancestry and dirty-tree checks, a real
`git worktree add` test fixture, and a documented decision about the one check that
deliberately stays anchored on `project_root`.
</objective>

**Assumption A3 (from 18-RESEARCH.md), adopted by this plan.**

`is_self_dogfood_workspace` stays `project_root`-scoped rather than switching to
`execution_root`.

*Reasoning:* it answers "is this workspace DevFlow's own repo at all", not "is the
binary stale relative to tree X". A git worktree shares the same tracked files as the
commit it is checked out to, so in practice both roots agree.

*Risk if wrong:* if a PLAN mid-flight modified the root `Cargo.toml`'s `members` array
on the feature branch, the two roots could disagree and the self-dogfood hard block
could mis-fire or fail to fire. Judged unlikely but not impossible. Task 2 requires
this choice to be documented in the source with its reasoning, so a future reader
encounters the decision rather than rediscovering it as a bug.
</flagged_assumptions>

<threat_model>
## Trust Boundaries

Local-only developer CLI. The boundary at issue is between the binary that is running
and the source tree it claims to represent.

| Boundary | Description |
|----------|-------------|
| compiled binary provenance → pipeline decisions | a binary's embedded commit is the only evidence that the code driving a phase matches the code under review |
| worktree branch → main checkout | the code under test lives on the worktree branch while DevFlow's bookkeeping lives in the main checkout; conflating them is the defect |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-22 | Tampering | `enforce_build_staleness` evaluated root | high | mitigate | A stale binary silently re-running pre-fix logic and reporting success is evidence tampering — the binary misrepresents which source produced the result. This already happened: Round 4 CR-01, where a binary two hours behind the worktree branch classified as `Ahead` and re-emitted a false changelog heading. Mitigated by evaluating `execution_root` so the check answers the question it was written to answer. |
| T-18-23 | Repudiation | blocking message naming the wrong tree | medium | mitigate | A block that names the main checkout when the worktree was evaluated sends the operator to rebuild against the wrong HEAD, which is what made the original incident hard to diagnose. Task 2 requires the message to name `execution_root` and to state whether a worktree was in play. |
| T-18-24 | Elevation of Privilege | staleness block becoming approvable | high | mitigate | The self-dogfood block is deliberately NOT an approvable gate — approving it would reintroduce the Phase 16 false-evidence incident. This plan changes only which tree is evaluated; `staleness_outcome` and the `Block` disposition are explicitly left untouched, and the prohibition forbids relaxing any `Block` to a `Warn`. |
| T-18-25 | Information Disclosure | `execution_root.display()` in the block message | low | accept | The blocking message is a terminal error shown to the operator whose machine it is, not a persisted artifact. It is NOT written into `events.jsonl`: the existing `events::emit` call passes the message through `truncate_reason`, and this plan does not add a path to any new event payload. Distinct from the WR-02 class, where a path was persisted. |
| T-18-26 | Denial of Service | false-positive block on a non-worktree phase | medium | mitigate | If the `unwrap_or(project_root)` fallback were wrong, every non-worktree phase could start hard-blocking. Pinned by `staleness_without_worktree_is_unchanged` plus the requirement that all eight pre-existing staleness tests pass unmodified. |
</threat_model>
