<objective>
Fix the Layer 0 short-circuit that makes Validate unpassable when `external_verify` is
declared (18e). 17-03 removed `evaluate_layer0`'s `stage != Code` guard and added an
affirmative-success arm returning `verdict: None`. `evaluate_agent_result_inner`
returns immediately on any `Some(..)` from Layer 0, so Layer 1 â€” the ONLY carrier of a
verdict â€” is never read, and `advance()` computes `passed = false` at Validate. An
agent's explicit `verdict: pass` is discarded.

This is a regression introduced by this project's own Phase 17. It is masked today only
because no PLAN in this repo declares `external_verify` â€” which is also why the test
suite is silent about it.

Purpose: implement the binding operator decision of 2026-07-20 â€” gate only when
ambiguous, not on every declared `external_verify`. Advance when the probe and the
agent's verdict agree; gate for a human when they disagree or when no verdict arrived.

Output: verdict reconciliation in devflow-core, a three-way outcome type in the CLI,
and the combined-scenario test the research requires for 18d+18e together.
</objective>

<assumption_delta>
**Pitfall 3 (from 18-RESEARCH.md) â€” the code shape, resolved by this plan.**

The research offered two viable shapes for the three-way outcome and explicitly left the
choice to the planner: (a) route disagreement/no-verdict through a
`handle_stage_failure`-shaped immediate gate, or (b) add a reason-string flag on
`AgentResult` checked at the `advance()` dispatch site.

*Decision:* **neither exactly â€” a dedicated `ValidateOutcome` enum in the CLI**, with
the ambiguous arm reusing `handle_stage_failure`'s `[never-silent]` gate SHAPE but with
Validate-appropriate dispatch targets.

Reason-string flags were rejected: this codebase already carries one stringly-typed
control signal (the `review:` prefix convention for Ship rejections), and its own doc
comment at `is_ship_review_failure` calls that "an inherent limitation of the
agent-reported DEVFLOW_RESULT contract". Adding a second is a known-bad pattern here. An
enum is compile-checked, so a future variant is a compile error rather than a silent
fall-through.

Routing literally through `handle_stage_failure` was also rejected: its Advance and
LoopBack arms both relaunch the SAME stage, which is wrong for an ambiguous Validate.
The human being asked to adjudicate needs Advance to mean "the verdict is right, go to
Ship" and LoopBack to mean "there are gaps, return to Code" â€” which is exactly the
dispatch `handle_validate_outcome`'s existing gated branch already performs.

*Tradeoff:* changing `handle_validate_outcome`'s signature from `passed: bool` to
`outcome: ValidateOutcome` touches all five call sites (two in `advance()`, three in the
test module). That churn is accepted because the boolean is precisely what cannot
represent the operator's three-way decision, and leaving it in place would force one of
the two rejected workarounds.
</assumption_delta>

<flagged_assumptions>
**UNRESOLVED edge probe â€” 18e (`unclassified`).**

The deterministic edge probe classified 18e's requirement as `unclassified` and it
therefore remains **unresolved**. Per protocol it is neither auto-backstopped nor
silently dropped; it is surfaced here for a human to adjudicate at verification time.

*What the planner could not settle:* the probe generated no applicable category
(boundary / empty / ordering / precision / idempotency / concurrency) for "Layer 0
short-circuit makes Validate unpassable when `external_verify` is declared", because
the requirement describes a control-flow defect rather than a data operation with an
input domain. The nearest genuine edge â€” what happens when Layer 0 affirmatively
succeeds at a stage OTHER than Validate â€” is deliberately out of scope here: 17-03
made Layer 0 run at every stage on purpose, and only Validate carries a verdict.

*Assumption made:* verdict reconciliation applies at `Stage::Validate` only. At every
other stage an affirmative Layer 0 success keeps `verdict: None` and its existing
meaning, unchanged from current behavior.

*Risk if wrong:* if a future stage gains verdict semantics, the reconciliation will not
apply to it and that stage will silently inherit today's discard behavior. Low today
(no other stage reads `verdict`); flagged so it is not rediscovered as a bug.
</flagged_assumptions>
