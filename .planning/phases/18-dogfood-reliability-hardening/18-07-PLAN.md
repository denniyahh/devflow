<objective>
Fix the preflight gate wedge (18f). In `run_preflight`, both `GateAction::Advance` and
`GateAction::LoopBack` call `launch_stage` again, which re-enters `run_preflight` and
re-runs the SAME check. Both production checks are deterministic, idempotent predicates
that a gate approval cannot change. `Gates::cleanup` deletes the operator's response
first, so the second identical failure writes a new gate and polls
`gate_timeout_secs()` — seven days — then returns an error leaving `gate_pending: true`
persisted.

Purpose: an operator who approves a gate expects the pipeline to proceed. Today
approval produces a seven-day hang. This is distinct from the double-launch bug 17-08
fixed, and its `Ok(bool)` contract must be preserved while fixing this one.

Output: a `launch_stage` split that lets the Advance arm skip the already-adjudicated
check, a persisted retry ceiling as the backstop, and an unconditionally-failing test
adapter that can actually reproduce the wedge.
</objective>

<assumption_delta>
**Open Question 2 (from 18-RESEARCH.md) — resolved by this plan.**

*Question:* where does the "bound the recursion regardless" backstop counter live — a
new `State` field, or a recursion-depth parameter?

*Decision:* **a persisted `State.preflight_retries: u32` field**, with a new
`MAX_PREFLIGHT_RETRIES` constant in `mode.rs` alongside the existing ceilings. This
adopts the researcher's recommendation.

*Reasoning:* the wedge CONTEXT.md documents spanned separate `devflow` invocations —
the monitor died and was manually recovered — which a purely in-recursion depth
parameter would not survive. It would reset to zero on every new process and therefore
fail to bound the exact incident it was written for. Persisting also matches the
existing `consecutive_failures` / `infra_failures` pattern, so there is one convention
for "bounded fault counters" rather than two.

*Tradeoff, stated explicitly:* a persisted counter needs an explicit reset discipline
that an in-recursion counter would get for free. A phase that hits two transient
preflight failures early and then runs healthily for days would otherwise abort on a
third, unrelated failure much later. This plan therefore resets `preflight_retries` to
0 on every preflight PASS and on every human Advance. That reset is a correctness
requirement of the chosen shape, not an optimization, and it is pinned by its own test.

*Second decision — the skip mechanism.* The skip is implemented by splitting
`launch_stage` into `launch_stage` (preflight, then delegate) and `launch_stage_inner`
(everything after preflight), and having the Advance arm call `launch_stage_inner`
directly. This is deliberately NOT a flag on `State` and NOT a parameter threaded
through `launch_stage`: a flag would be a persisted bypass that could outlive the stage
it was granted for, which the research explicitly names as an anti-pattern. Calling a
different function is inherently scoped to that one call — there is no bypass state
that can leak to a later stage, because there is no bypass state at all.
</assumption_delta>

<flagged_assumptions>
**UNRESOLVED edge probe — 18f (`unclassified`).**

The deterministic edge probe classified 18f's requirement as `unclassified` and it
therefore remains **unresolved**. Per protocol it is neither auto-backstopped nor
silently dropped; it is surfaced here for a human to adjudicate at verification time.

*What the planner could not settle:* the probe generated no applicable category for
"approving a preflight gate re-runs the identical failing check and wedges", because
the requirement describes a control-flow recursion defect rather than an operation
over an input domain. The nearest real edge — the value of `MAX_PREFLIGHT_RETRIES` —
is a policy choice with no derivable correct answer.

*Assumption made:* `MAX_PREFLIGHT_RETRIES = 3`, matching `MAX_CONSECUTIVE_FAILURES`
rather than the more lenient `MAX_INFRA_FAILURES = 5`. Reasoning: a failing preflight
is a readiness problem the operator is actively being asked about, not a transient
cloud blip, so the tighter ceiling is the better fit. This is a judgement call, not a
derived value — an operator who wants it looser should say so.
</flagged_assumptions>
