<objective>
Make the Code↔Validate safety gate reachable (18d). `transition()` unconditionally
zeroes `consecutive_failures`, and the Code→Validate hop is crossed on every single
retry cycle — so the counter oscillates 0↔1 and `MAX_CONSECUTIVE_FAILURES = 3` is
unreachable for the exact loop it was written to bound. Observed live across three
cycles; under `--mode auto` this loops indefinitely while `status` shows a healthy
alternating pipeline.

Purpose: this is the mechanism that turns a stuck phase into an invisible one. The
ceiling already exists and the gate policy already exists; only the reset is wrong.

Output: a pure `mode.rs` predicate deciding when the reset fires, wired into
`transition()`, with the infra counter's behavior provably untouched.
</objective>

<assumption_delta>
**Open Question 1 (from 18-RESEARCH.md) — resolved by this plan.**

*Question:* should 18d's fix live as a new pure predicate in `mode.rs` (consistent with
`should_gate`/`should_auto_loop`), or stay a local conditional inside `transition()`?

*Decision:* **new pure predicate in `mode.rs`**, named
`transition_resets_consecutive_failures(from: Stage, to: Stage) -> bool`. It is a free
function, not a `Mode` method — unlike `should_gate` and `should_auto_loop`, the reset
rule does not depend on the execution mode, and forcing it onto `Mode` would imply a
mode-dependence that does not exist.

*Tradeoff, stated explicitly:* the cost is a devflow-core ↔ devflow-cli round trip for
what is otherwise a one-line conditional, plus a `Stage`-pair rule living in a module
named for `Mode`. The benefit is that the reset rule becomes directly unit-testable
over its whole input domain with no repository, no state file, and no process spawn —
the same property `mode.rs`'s existing predicate tests already have — and it sits
beside the ceiling constant it protects. `mode.rs`'s `MAX_INFRA_FAILURES` doc comment
already documents the reset's INTENT even though the code lives in `main.rs`; this
moves the decision to where the knowledge already was. If a future reader disagrees,
inlining the predicate back into `transition()` is a mechanical change.
</assumption_delta>

<threat_model>
## Trust Boundaries

Local-only developer CLI. The boundary at issue is between the pipeline's own
self-reported progress and the safety ceiling that is supposed to interrupt it.

| Boundary | Description |
|----------|-------------|
| agent-driven stage loop → operator attention | an unbounded auto-loop consumes wall-clock time, agent quota, and operator trust while `status` reports a healthy alternating pipeline |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18-13 | Denial of Service | `transition()` / `MAX_CONSECUTIVE_FAILURES` | high | mitigate | Local resource and time exhaustion via an infinite retry loop that never reaches its own ceiling — observed live across three cycles. This is the V1 (ASVS Architecture) unbounded-retry class named in 18-RESEARCH.md's security domain. Mitigation is to make the EXISTING bound reachable via `transition_resets_consecutive_failures`, not to invent a new one. |
| T-18-14 | Tampering | `infra_failures` reset scope | high | mitigate | Widening or narrowing the infra counter's reset while fixing the consecutive counter would silently convert `MAX_INFRA_FAILURES` from a stuck-loop bound into a phase-lifetime bound — the exact regression 17-06 was written to prevent. Mitigated by leaving the infra reset unconditional and requiring the pre-existing `transition_resets_infra_failures` test to pass unedited, asserted as its own acceptance criterion. |
| T-18-15 | Denial of Service | `consecutive_failures` u32 overflow | medium | mitigate | Now that the counter genuinely accumulates, a long-running stuck loop could wrap it to 0 and restore the unreachable-ceiling bug in a slower, harder-to-diagnose form. Mitigated with `saturating_add`, matching the convention `infra_failures` already documents, and pinned by `consecutive_failures_increment_saturates`. |
| T-18-16 | Repudiation | gate firing without a recorded cause | low | accept | `transition()` already emits a `transition` event and `run_gate` already records gate lifecycle events; this plan adds no new silent branch. |
