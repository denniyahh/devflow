<objective>
Close WR-11's silent-halt bug and give the Ship stage a real failure branch. Today
`advance()`'s catch-all `_ => Err(...)` arm returns an error but writes no gate and sends
no notification for any non-Validate stage failure (Define/Plan/Code/Ship), leaving state
dirty with `gate_pending: false` — nothing ever surfaces the halt. This plan routes every
non-Validate failure through a gate + a new pluggable notify hook, distinguishes Ship
`AgentFailed` (gate+notify) from `ReviewFailed` (loop back to Code), and makes the gate
poll timeout env-configurable.

Purpose: unattended runs must always surface a halt or a gate — silence is the failure
mode that makes dogfooding (13e) impossible. This is the architecturally riskiest chunk
and 13e's dogfood acceptance depends on it, so it is front-loaded into Wave 1.
Output: `fire_gate_notify` in gates.rs; `handle_stage_failure`, `handle_ship_failure`,
and `gate_timeout_secs()` in main.rs; notify wired into `run_gate`; failing-then-passing
unit tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator env → notify command | DEVFLOW_GATE_NOTIFY_CMD is operator-configured (trusted); gate context passed to it may contain agent-generated text (untrusted) |
| agent stdout → advance() reason | The `reason` string driving ReviewFailed/AgentFailed originates from agent output (untrusted) but is only string-matched, never executed |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-13-01 | Tampering | fire_gate_notify shell-out | high | mitigate | Never interpolate phase/stage/context into the `sh -c` string; pass via `.env()` vars (DEVFLOW_GATE_PHASE/STAGE/CONTEXT), following the WR-01 argv-not-shell precedent. Test asserts no string concatenation of context into the command. |
| T-13-02 | Denial of Service | notify hook blocking the loop | medium | mitigate | Fail-soft: fire_gate_notify warns and returns on non-zero exit or spawn failure, never propagates an error into run_gate. |
| T-13-03 | Denial of Service | env-configurable gate timeout set absurdly low/high | low | accept | Operator-controlled single-user tool; a bad DEVFLOW_GATE_TIMEOUT_SECS only affects the operator's own run. Falls back to 7 days on parse failure. |
| T-13-04 | Spoofing | agent emits `review:`-prefixed reason to force loop-back | low | accept | Local single-operator trust boundary; the agent is already fully trusted to run under `--dangerously-skip-permissions`. Loop-back to Code is a safe action (no merge/ship occurs). |

No new external packages are added by this plan (no npm/pip/cargo install) — package-legitimacy checkpoint not applicable.
</threat_model>
