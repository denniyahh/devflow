<objective>
Deliver unit 21b (D-04, D-05): add a detection-only `doctor` check that compares the version/outcome claims in `.planning/ROADMAP.md` (`## Shipped`) and `.planning/STATE.md` (`## Completed`) against the repo's git tags, and flags drift — the exact class of bug this session hit (STATE/ROADMAP claiming Phase 18 unreleased after v1.5.0 shipped). It never rewrites prose (D-04); it integrates as a new finding in the existing `doctor` path so human and `--json` output stay consistent (D-05).

The load-bearing risk is noise: `ROADMAP.md`'s `## Shipped` and `STATE.md`'s `## Completed` carry version ranges (`0.1.0–0.6.0`), em-dash placeholders (`—`), and duplicate/legacy versions (Phase 9 and Phase 11 both claim `1.2.0`; Phases 6/7 claim `1.0.0` but no `v1.0.0` tag exists — tags start at `v1.0.1`). A naive per-row check floods `doctor` with Problem-severity findings on every run — the exact alert-fatigue failure 999.14 exists to prevent. Scope the check to cells matching `^v?\d+\.\d+\.\d+$`, skip the rest, and downgrade pre-v1.5.0 mismatches to Warn/informational.

Assumptions surfaced: (1) finding type is a NEW sibling `PlanningDocFinding` (not `PhaseFinding`), because most claims are about already-shipped phases with no active `state-NN.json`/`PhaseFacts` (RESEARCH Open Q1); (2) the noise cutoff is a fixed `v1.5.0` boundary (first consistently-tagged release) rather than a config knob. This plan is Wave 3; it depends on 21-02 (shared `commands.rs`).

Purpose: make devflow catch its own planning-doc drift instead of an operator finding it weeks later.
Output: a third, detection-only `doctor` finding set, in both text and `--json`.
</objective>
