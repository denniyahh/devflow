# Phase 11: Branching & Merge Scheme + Dev Container

**Status:** Requirements defined | **Priority:** HIGH | **Target:** TBD

## Goal

Complete DevFlow's branching and merge implementation: wire up `feature_finish` and `release_finish`, add merge guard rails, and optionally ship a dev container.

> **Note:** Conventional commits (`feat:`, `fix:`, etc.) were deprecated in June 2026. Phase 11 replaces them with DevFlow's native branching conventions where phase identity drives the git history.

---

## 11a ‚Äî Wire `feature_finish` ‚Üí `devflow finish`

`feature_finish()` exists in `git.rs` but is never called by the CLI. Wire it up as a new command.

- [ ] New command: `devflow finish [--phase N] [--push]`
- [ ] Guard rail: reject if not on a `feature/*` branch (matches `git_flow.feature_prefix`)
- [ ] Action: merge `feature/phase-NN` ‚Üí `develop` with `--no-ff`, delete local feature branch
- [ ] `--push`: push develop to origin, delete remote feature branch
- [ ] Integration: called automatically by state machine after `Executing` step completes (when agent succeeds)
- [ ] Test: `devflow finish` merges feature into develop, deletes branch, develop has the commits

---

## 11b ‚Äî New `devflow release` Command

`release_finish()` exists in `git.rs` but is never called by the CLI. Add a separate command for production releases ‚Äî decoupled from merging to develop.

- [ ] New command: `devflow release [--push]`
- [ ] Guard rails:
  - Must have a valid last-ship record (`.devflow/last-ship.json`)
  - Must be on `develop` or the release branch
  - Reject if no release branch exists
- [ ] Action (calls `release_finish(version)`):
  1. Checkout `main`
  2. Merge `release/X.Y.Z` into `main` with `--no-ff`
  3. Tag `vX.Y.Z` on main
  4. Checkout `develop`
  5. Merge `release/X.Y.Z` into `develop` with `--no-ff` (brings version bump + any final fixes back)
  6. Delete local `release/X.Y.Z` branch
- [ ] `--push`: push main + develop + tags to origin, delete remote release branch
- [ ] Test: `devflow release` merges release ‚Üí main, tags correctly, merges ‚Üí develop, deletes branch

## 11c ‚Äî Merge Guard Rails (`git_flow.enforce`)

New config option to validate merge topology before any git-flow operation.

- [ ] New config field: `git_flow.enforce` (bool, default `true`)
- [ ] Guard checks:
  - `devflow finish` validates source branch matches `feature_prefix`
  - `devflow ship` validates current branch is a feature branch (not main/develop directly)
  - `devflow release` validates last-ship record exists, release branch exists, target topology is correct
- [ ] When `enforce: false`, all guard rails are skipped (opt-out for non-standard workflows)
- [ ] Config schema: add `enforce` field to `GitFlowConfig`, `parse_config()`, `default_config_yaml()`, `to_yaml()`
- [ ] Test: guard rails reject wrong-source merges; `enforce: false` allows them

## Updated Ship Flow

```
devflow start    ‚Üí feature/phase-NN (from develop) ‚Üí agent works
devflow finish   ‚Üí feature/phase-NN merged into develop, branch deleted
devflow ship     ‚Üí release/X.Y.Z (from feature HEAD), version bump, PR to develop
 [PR review + merge into develop]
devflow release  ‚Üí merge release ‚Üí main, tag vX.Y.Z, merge ‚Üí develop, cleanup
devflow confirm  ‚Üí CHANGELOG + ROADMAP update, clears ship record
```
