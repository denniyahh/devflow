# Phase 11: Branding & Merge Scheme + Dev Container

**Status:** Requirements defined | **Priority:** HIGH | **Target:** TBD

## Goal

Replace conventional commits with DevFlow's native git-flow completion: wire up `feature_finish` and `release_finish`, add merge guard rails, introduce phase-branded commit conventions, and optionally ship a dev container.

> **Note:** Conventional commits (`feat:`, `fix:`, etc.) were deprecated in June 2026. Phase 11 replaces them with phase-branded commits and structured merge conventions.

## 11a â€” Wire `feature_finish` â†’ `devflow finish`

- [ ] New command: `devflow finish [--phase N] [--push]`
- [ ] Guard rail: reject if not on a `feature/*` branch (matches `git_flow.feature_prefix`)

## 11b â€” New `devflow release` Command

`release_finish()` exists in `git.rs` but is never called by the CLI. Add a separate command for production releases â€” decoupled from merging to develop.

- [ ] New command: `devflow release [--push]`
- [ ] Guard rails:
  - Must have a valid last-ship record (`.devflow/last-ship.json`)
  - Must be on `develop` or the release branch
  - Reject if no release branch exists
- [ ] Action (calls `release_finish(version)`):
  1. Checkout `main`
  2. Merge `release/X.Y.Z` into `main` with `--no-ff`
  3. Tag `vX.Y.Z` on main
  4. Checkout `develop`
  5. Merge `release/X.Y.Z` into `develop` with `--no-ff` (brings version bump + any final fixes back)
  6. Delete local `release/X.Y.Z` branch
- [ ] `--push`: push main + develop + tags to origin, delete remote release branch
- [ ] Test: `devflow release` merges release â†’ main, tags correctly, merges â†’ develop, deletes branch

## 11c â€” Merge Guard Rails (`git_flow.enforce`)

New config option to validate merge topology before any git-flow operation.

- [ ] New config field: `git_flow.enforce` (bool, default `true`)
- [ ] Guard checks:
  - `devflow finish` validates source branch matches `feature_prefix`
  - `devflow ship` validates current branch is a feature branch (not main/develop directly)
  - `devflow release` validates last-ship record exists, release branch exists, target topology is correct
- [ ] When `enforce: false`, all guard rails are skipped (opt-out for non-standard workflows)
- [ ] Config schema: add `enforce` field to `GitFlowConfig`, `parse_config()`, `default_config_yaml()`, `to_yaml()`
- [ ] Test: guard rails reject wrong-source merges; `enforce: false` allows them
